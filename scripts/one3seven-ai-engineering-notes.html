Building Real-World AI Systems in Production
The gap between "AI demo" and "production AI system" — and the specific patterns that bridge it.

The Core Lesson

Production AI systems fail differently than demos. In a demo, you control the inputs — you pick the document, you know the expected output, and you never test edge cases. In production, users send scanned images, handwritten notes, files named "final_FINAL_v3.pdf," and documents with zero machine-readable text. The system must handle all of them gracefully.

The engineering discipline for this is not "make the AI smarter." It is: design the system so failures are bounded, visible, and recoverable.

Key Production Patterns

1. Treat AI output as untrusted input

Every piece of AI-generated content should go through the same validation that user input goes through. If the AI returns a date, validate it's a real date before storing it. If it returns a list of events, validate the schema before rendering.

❌ Demo Thinking

Trust that Claude returns a valid date string in the timeline. Render it directly.

✓ Production Thinking

Validate every date extracted. Fall back to "Date not confirmed" if invalid. Never crash the render layer on bad AI output.

2. Non-blocking intelligence

Intelligence extraction (Phase 2B) must never block the core intake workflow. If Claude fails, the firm still sees the intake. If extraction times out, a retry can be scheduled. The base product is the chronology and documents — intelligence is an enhancement.

one3Seven Already Does This Right

Phase 2B is already non-blocking in the current architecture — `get-intake-intelligence` failure is caught and swallowed in `loadFirmLiveIntakeView`. The intelligence field is null, not an error state. This is the correct production pattern. Keep it.

3. Separate concerns: extraction vs. synthesis vs. rendering

Three distinct steps should never be merged into one AI call:

Raw Documents
→
Extract Facts
→
Persist to DB
→
Synthesize
→
Single Packet
→
Render

one3Seven has this right architecturally — `extract-document-facts` (per file) and `get-intake-intelligence` (synthesis) are separate Edge Functions. The gap is that synthesis and rendering diverge between the firm review screen and the PDF exporter.

4. Idempotency is mandatory

Any operation that touches the database must be safe to run twice. If a document is re-processed, it should produce the same result, not a duplicate. The `fact_extraction_status = 'completed'` guard in `extract-document-facts` is the right pattern — already present, must be preserved.

Applied to one3Seven — Action Items

1. The "Organizing Records" stuck state is a violation of this principle. Organization must be idempotent and resumable — if interrupted, it should continue from where it stopped, not restart from zero and risk duplicates.

2. The firm review screen and PDF exporter must call the same synthesis function, not implement their own versions of it. One source, two renderers.

3. All timeline event dates extracted by AI must be validated before storage — not just before rendering.

Structured Outputs for Reliable Applications
Why enforcing a schema on AI output is the single most important reliability upgrade available — and how to do it.

The Core Lesson

Free-form AI text is unpredictable. Even with a perfect prompt, the model may return "the date was November 26th" in one call and "Nov 26, 2025" in the next. Structured outputs force the model to return a specific JSON schema — guaranteed. No parsing, no string manipulation, no guessing.

OpenAI's Structured Outputs feature (and Anthropic's equivalent) uses JSON Schema enforcement at the API level — not just in the prompt. The model cannot return a malformed response. This eliminates an entire category of production bugs.

The Pattern

Unstructured — What one3Seven Does Today (Phase 2A deterministic path)
// Prompt asks Claude to write a paragraph summarizing events
// Output: "Marcus Rivera filed an HR complaint on November 26th, 2025..."
// Problem: parsing prose to extract dates, names, events is fragile

const result = await anthropic.messages.create({
  model: 'claude-sonnet-4-5',
  messages: [{ role: 'user', content: prompt }]
  // No schema enforcement — model returns whatever it wants
});
Structured — What one3Seven Should Do (Phase 2B intelligence path)
// Define the exact shape of what you want back
const DocumentFactsSchema = {
  type: "object",
  properties: {
    complaint_date:      { type: "string", description: "ISO 8601 date or null" },
    complaint_topic:     { type: "string" },
    warning_date:        { type: "string", description: "ISO 8601 date or null" },
    warning_reason:      { type: "string" },
    termination_date:    { type: "string", description: "ISO 8601 date or null" },
    termination_reason:  { type: "string" },
    key_people:          { type: "array", items: { type: "string" } },
    key_quotes:          { type: "array", items: { type: "string" } },
    dates_present:       { type: "array", items: { type: "string" } },
    source_confidence:   { type: "string", enum: ["high", "partial", "low"] }
  },
  required: ["source_confidence"],
  additionalProperties: false
};

// Anthropic equivalent: structured output via tool use with strict schema
const result = await anthropic.messages.create({
  model: 'claude-sonnet-4-5',
  tools: [{ name: 'extract_document_facts', input_schema: DocumentFactsSchema }],
  tool_choice: { type: 'tool', name: 'extract_document_facts' },
  messages: [{ role: 'user', content: prompt }]
  // Model MUST return valid DocumentFacts — no free-form text possible
});

Why This Directly Solves the Consistency Problem

The live review screen and PDF export produce different outputs today because they both generate content from prose — either from stored prose summaries or from re-prompting the model. If both surfaces read from a single structured object stored in the database, they can never diverge.

Documents
→
Structured Extraction
→
document_facts (JSONB)
→
review_packet_v1
→

+

Applied to one3Seven — The Fix

The `extract-document-facts` Edge Function already uses structured tool-use calls to Claude — the `document_facts` JSONB field is already a structured object. The gap is downstream: the firm review screen and PDF exporter don't both read from that same object. They have independent rendering logic.

The fix is not to change extraction. It is to create `review_packet_v1` — a single synthesized object derived from `document_facts` — and have both rendering surfaces read from it.

Structured Output — All You Need to Know
The practical implementation guide — TypeScript types, Pydantic models, strict mode, and the tradeoffs.

The Core Lesson

Structured output is not just about JSON — it is about making the contract between your application and the AI model explicit and enforceable. When you define a TypeScript type for what you expect back, you are writing a specification. When you enforce it at the API level, the specification becomes a guarantee.

The review_packet_v1 Type Definition

This is what one3Seven's central intelligence object should look like. Every surface reads from this — nothing is generated independently:

TypeScript — review_packet_v1 (proposed)
interface TimingInterval {
  label:       string;           // "HR response", "Termination"
  days:        number;
  description: string;           // human-readable sentence
}

interface KeyQuote {
  file_name:   string;
  category:    string;
  quote:       string;
  confidence:  'high' | 'partial' | 'low';
}

interface TimelineEvent {
  id:             string;
  event_date:     string;      // ISO 8601 or human-readable range
  title:          string;
  category:       string;
  summary:        string;      // document-derived only
  worker_context: string | null;
  source_files:   string[];    // direct file names, not parsed from prose
  source_type:    'record_grounded' | 'worker_reported' | 'mixed';
}

interface ReviewPacketV1 {
  // Identity
  intake_id:             string;
  generated_at:          string;       // ISO 8601 timestamp
  schema_version:        '1';           // bump when shape changes

  // Summary
  overview:              string;        // one paragraph, deterministic
  matter_tags:           string[];

  // Timeline
  timeline:              TimelineEvent[];
  timeline_span:         string;        // "March 2022 – January 2026"
  timeline_count:        number;

  // Records
  record_count:          number;
  record_categories:     string[];
  missing_records:       string[];

  // Intelligence (Phase 2B — null until extraction runs)
  intelligence: {
    timing_intervals:    TimingInterval[];
    key_quotes:          KeyQuote[];
    confirmation_items:  string[];
    extracted_facts:     Record<string, unknown>;
  } | null;
}

// Both of these read from ReviewPacketV1 — they never generate independently
function renderFirmReviewScreen(packet: ReviewPacketV1): JSX.Element { ... }
function buildFirmPDF(packet: ReviewPacketV1): Uint8Array { ... }
The Critical Insight

`schema_version: '1'` on the packet means you can detect when old packets need regeneration. When you change the shape in a future sprint, old packets with version '0' or without the field can be flagged for re-synthesis without breaking anything currently rendering.

Applied to one3Seven — What This Unlocks

Once `ReviewPacketV1` exists as the single source, the PDF export function `buildFirmIntakeSummaryPDF` in `firmIntakeSummaryDownload.ts` stops maintaining its own version of the chronology, record count, and clarifications. It reads `packet.timeline`, `packet.record_count`, `packet.intelligence.confirmation_items`. Screen and PDF are guaranteed identical because they share one object.

What Are AI Evaluations?
Human evals, programmatic evals, and LLM-as-judge — when to use each and why one3Seven needs all three.

The Core Lesson

An evaluation is the answer to: "How do I know the AI did it correctly?" Without evals, you rely on vibes — it seemed right, the demo looked good, no one complained yet. With evals, you have a system that catches regressions before they reach production.

There are three types of evals. Each catches different things.

The Three Eval Types

Eval TypeHow It WorksBest ForCost
Programmatic Code checks the output — schema validation, regex match, exact value, count Schema correctness, required fields present, dates parse, no banned phrases Free, instant, fully automated
LLM-as-Judge A second AI model reads the output and scores it against criteria Tone accuracy, legal-language safety, summary quality, completeness API cost per eval, fast to set up
Human A real person reviews output and marks it correct / incorrect / partially correct Qualitative accuracy, attorney usefulness, trust calibration Slow and expensive — use sparingly

The Crestline Problem — Diagnosed as an Eval Gap

The live review screen and PDF showed different content for the same intake. That is an eval failure — specifically a missing programmatic eval that should check:

Programmatic Eval — Consistency Check (should exist today)
async function evalPacketConsistency(intakeId: string) {
  const liveData = await loadFirmLiveIntakeView(intakeId);
  const pdfData  = await buildFirmIntakeSummaryPDF(intakeId);

  // These should never differ — if they do, it's a bug
  assert(liveData.events.length    === pdfData.timelineCount,   'timeline count mismatch');
  assert(liveData.files.length     === pdfData.recordCount,     'record count mismatch');
  assert(liveData.missing.length   === pdfData.gapCount,        'gap count mismatch');

  // Every clarification in the PDF should appear in the live view
  for (const item of pdfData.confirmationItems) {
    assert(liveData.intelligence?.confirmationNeeded.includes(item),
      `PDF item not in live view: ${item}`);
  }
}

The LLM-as-Judge Pattern for one3Seven

After extraction, a second Claude call evaluates the output:

LLM-as-Judge — Legal Language Safety Check
const judgePrompt = `
You are reviewing AI-generated intake summary text for an employment law intake platform.
The platform must NEVER produce legal conclusions, claim assessments, or outcome predictions.

Review this text and return JSON:
{
  "safe": boolean,
  "violations": string[],  // exact phrases that are legal conclusions
  "reason": string
}

Text to review:
"${generatedText}"
`;

// Run after every synthesis pass — before storing to database
// If safe === false: do not store, log violation, fall back to deterministic output
Applied to one3Seven — Three Evals to Build

1. Programmatic — Schema validation. Every `document_facts` write must pass a JSON schema check before being stored. Dates must parse. Required fields must be present.

2. Programmatic — Consistency check. Before any deployment, run `evalPacketConsistency` against the Crestline intake and confirm the live view and PDF agree on all counts.

3. LLM-as-Judge — Language safety. The `BANNED_IN_OUTPUT` regex is a programmatic safety check. An LLM judge adds a second layer that catches near-misses the regex doesn't pattern-match.

Intro to LLM Evaluation
Building a golden dataset and running repeatable evals — the shift from "it seemed right" to "it passed."

The Core Lesson

An eval framework needs three things: a golden dataset (known inputs with known correct outputs), a scoring function (how you measure correctness), and a harness (code that runs the eval and reports results). Without all three, you can't measure whether a prompt change made things better or worse.

The Golden Dataset for one3Seven

You already have one — you just don't know it yet. The Crestline intake (f50f8bfd) with 12 completed extractions is your first golden record. The Marcus Rivera demo intake is your second. Together they define the baseline. Every future model upgrade, prompt change, or pipeline change should be validated against both.

Golden Dataset Structure — one3Seven Evals
// evals/golden-dataset.ts

export const goldenIntakes = [
  {
    id:          'crestline-logistics',
    description: 'Wage theft + retaliation, 12 documents, fully extracted',
    expected: {
      timeline_count:      6,      // must produce at least 6 events
      record_count:        12,     // must reflect all uploaded files
      has_intelligence:    true,   // Phase 2B must run
      timing_intervals:    2,      // complaint→warning, complaint→termination
      key_quotes_count:    3,      // at minimum
      banned_phrases:      [],     // zero legal conclusions in any output field
    }
  },
  {
    id:          'marcus-rivera-demo',
    description: 'Demo intake — curated, 11 documents, known correct output',
    expected: {
      timeline_count:      6,
      record_count:        11,
      has_intelligence:    true,
      timing_intervals:    2,
      key_quotes_count:    3,
      banned_phrases:      [],
    }
  }
];

The Scoring Function

Scoring for one3Seven is not "did the AI write good prose." It is deterministic and measurable:

MetricPass ConditionFail Action
Schema validityAll required fields present, correct typesBlock storage, alert
Event count≥ expected count for this intakeFlag for review
Date validityAll date fields parse as valid dates or are nullStore as null, flag
Banned phrase checkZero regex matches in any generated fieldBlock output, fallback
ConsistencyLive view and PDF agree on all countsBlock deploy
Intelligence renderPanel populates for full-access intakes with extracted factsInvestigate pipeline
The Mindset Shift

The question stops being "did Claude seem to get it right?" and becomes "did the output pass the eval suite?" That shift is what separates a production intelligence layer from a demo intelligence layer. The Crestline test was an accidental human eval. Build the programmatic version so it runs on every deploy.

Applied to one3Seven — Minimum Viable Eval Suite

You do not need a full eval framework today. You need three things in a single test file: (1) a schema validator that runs against `document_facts` shape, (2) a consistency check that compares live view counts to PDF counts for the Crestline intake, (3) a banned-phrase check that runs the `BANNED_IN_OUTPUT` regex against every text field in the intelligence object. Run it in CI or manually before every deploy. That's it for v1.

How to Systematically Setup LLM Evals
Metrics, unit tests, and LLM-as-judge — the system that makes your intelligence layer trustworthy at scale.

The Core Lesson

Systematic LLM evals start with defining what "correct" means for your specific use case, not for AI in general. For one3Seven, "correct" has a precise and narrow definition: the output is an accurate, source-linked, legal-conclusion-free organization of the records the worker submitted. That definition is the foundation of every eval metric.

Four Layers of Systematic Evals

Layer 1 — Unit Tests (fastest, cheapest)

Test individual functions in isolation with known inputs and expected outputs. No AI calls required.

Unit Test Examples for one3Seven
// Test: date normalization never crashes on edge-case AI output
expect(normalizeEventDate("November 26th, 2025")).toBe("2025-11-26");
expect(normalizeEventDate("unclear")).toBeNull();
expect(normalizeEventDate("")).toBeNull();

// Test: banned phrase regex catches all prohibited patterns
expect(containsBannedPhrase("retaliation pattern detected")).toBe(true);
expect(containsBannedPhrase("timing relationships identified")).toBe(false);
expect(containsBannedPhrase("wrongful termination case")).toBe(true);

// Test: timeline count in live view matches PDF builder input
expect(buildPDFInput(mockPacket).timelineCount).toBe(mockPacket.timeline.length);

Layer 2 — Integration Tests (medium cost)

Use a real intake with known expected output. Run the full pipeline. Compare to golden dataset.

Layer 3 — LLM-as-Judge (flexible, cost per call)

A separate Claude call reviews the output for qualitative correctness that code can't check.

Layer 4 — Human Eval (high cost, low frequency)

An attorney reviews the output and rates its usefulness. Run this once per major version, not per deploy.

The Regression Prevention Rule

⚠ The Most Important Pattern

Every time you find a bug manually — like the Crestline consistency mismatch — write a test that would have caught it before it reached production. The test suite should grow with every discovered failure. A bug that has been found once must never be found again through manual testing.

Applied to one3Seven — The Complete Picture

You have been running human evals manually — opening the Crestline intake, reading the output, comparing to the PDF, noticing the mismatch. That process works but doesn't scale and doesn't prevent regressions. The programmatic eval suite converts your manual review process into automated tests that run every time something changes. The goal is: the Crestline intake always renders the same thing on the firm review screen as in the PDF, and you know it before the deploy ships, not after a firm notices.
